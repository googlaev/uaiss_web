## 1. Общие сведения

### 1.1. Наименование системы

Наименование: «Система управления экзаменами и статусами сотрудников» (далее — Система).
Условное обозначение: **UAISS Web**.

### 1.2. Назначение системы

Система предназначена для предоставления сотрудникам организации удобного мобильно-адаптивного веб-интерфейса для выполнения следующих задач:

- Регистрации и просмотра записей о сдаче экзаменов по охране труда и промышленной безопасности.
- Управления статусами сотрудников (больничный, командировка, отпуск).
- Просмотра общей статистики и актуальных сводок по подразделению.
- Получения уведомлений об истечении срока действия экзаменов.

Система является веб-версией существующего Telegram-бота @virtualassistantUAISS_bot и работает с той же базой данных SQLite.

### 1.3. Основание для разработки

Документ разработан на основании:

- Инструкции по использованию Telegram-бота @virtualassistantUAISS_bot.
- Структуры базы данных `exams.db`.
- Требований к мобильно-ориентированному интерфейсу.

## 2. Описание базы данных

Система использует существующую базу данных SQLite (файл `exams.db`). Ниже приведено полное описание схемы.

### 2.1. Таблица `users` — Пользователи

| Поле | Тип | Описание |
|---|---|---|
| `user_id` | INTEGER (PK) | Уникальный внутренний идентификатор пользователя |
| `telegram_id` | INTEGER | ID пользователя в Telegram |
| `full_name` | TEXT | Полное имя (ФИО) сотрудника |
| `login` | TEXT | Логин для входа в веб-приложение *(добавить при разработке)* |
| `password_hash` | TEXT | Хэш пароля (bcrypt) *(добавить при разработке)* |
| `role` | TEXT | Роль: `employee` / `admin` *(добавить при разработке)* |

### 2.2. Таблица `exam_types` — Типы экзаменов

| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER (PK) | Уникальный идентификатор типа |
| `name` | TEXT | Краткое наименование (А1, Б21, ОТ и т.д.) |
| `duration` | INTEGER | Срок действия экзамена в месяцах |
| `emoji` | TEXT | Эмодзи-иконка для отображения в интерфейсе |

**Текущие типы экзаменов в системе:**

| Тип | Срок действия | Иконка |
|---|---|---|
| А1 | 60 месяцев (5 лет) | 🔬 |
| Б21 | 60 месяцев (5 лет) | 📋 |
| Б26 | 60 месяцев (5 лет) | 📊 |
| ОТ | 36 месяцев (3 года) | 🦺 |
| ОТБ | 36 месяцев (3 года) | 🛡️ |
| СИЗ | 36 месяцев (3 года) | 🥽 |
| ПП | 36 месяцев (3 года) | 🚒 |
| ЭБ | 12 месяцев (1 год) | 💡 |
| ПТМ | 6 месяцев | 🔥 |

### 2.3. Таблица `exams` — Записи о сдаче экзаменов

| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER (PK) | Уникальный идентификатор записи |
| `user_id` | INTEGER (FK) | Ссылка на пользователя (`users.user_id`) |
| `name` | TEXT | Тип экзамена (краткое наименование, например `ПТМ`) |
| `date` | TEXT | Дата сдачи в формате `ДД.ММ.ГГГГ` |
| `duration` | TEXT | Срок действия в месяцах (строка) |
| `notification_sent` | BOOLEAN | Флаг: общее уведомление отправлено |
| `month_notification_sent` | BOOLEAN | Флаг: уведомление за месяц отправлено |
| `week_notification_sent` | BOOLEAN | Флаг: уведомление за неделю отправлено |
| `exam_day_notification_sent` | BOOLEAN | Флаг: уведомление в день экзамена отправлено |
| `end_day_notification_sent` | BOOLEAN | Флаг: уведомление в день окончания отправлено |

### 2.4. Таблица `user_status` — Статусы сотрудников

| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER (PK) | Уникальный идентификатор записи |
| `user_id` | INTEGER (FK) | Ссылка на пользователя (`users.user_id`) |
| `status` | TEXT | Тип статуса с эмодзи (`🤒 Больничный`, `✈️ Командировка`, `🏖️ Отпуск`) |
| `start_date` | TEXT | Дата начала в формате `ДД.ММ.ГГГГ` |
| `end_date` | TEXT | Дата окончания (`NULL` для открытого больничного) |

> 💡 Больничный открывается с `end_date = NULL` и закрывается отдельным запросом, проставляющим `end_date = текущая дата`.

## 3. Архитектура системы

### 3.1. Общая архитектура

Система строится по принципу клиент-серверного SPA с адаптивным интерфейсом:

- **Frontend** (React / Vue.js) — мобильно-ориентированный интерфейс, PWA-приложение.
- **Backend** (REST API, Python FastAPI) — бизнес-логика, работа с БД, управление сессиями.
- **База данных** — SQLite (`exams.db`), общая с Telegram-ботом.
- **Уведомления** — задачи по расписанию (APScheduler) для проверки истекающих экзаменов.

### 3.2. Стек технологий

| Слой | Технология | Обоснование |
|---|---|---|
| Frontend | React 18 + TypeScript | Компонентный подход, PWA-поддержка, мобильная адаптивность |
| UI Kit | Tailwind CSS + shadcn/ui | Мобильные компоненты, тёмная тема, быстрая разработка |
| Backend | Python FastAPI | Нативная работа с SQLite, асинхронность, автодокументация |
| БД | SQLite (exams.db) | Существующая база, совместимость с Telegram-ботом |
| Авторизация | Простая веб-форма (логин + пароль) | Вход через стандартную форму, сессия хранится в cookie |
| Планировщик | APScheduler (Python) | Фоновые задачи: уведомления об экзаменах |
| Деплой | Docker + nginx | Контейнеризация, обратный прокси |

### 3.3. REST API — эндпоинты

Базовый URL: `/api/v1`
Все защищённые эндпоинты проверяют наличие активной сессии через cookie.

#### 3.3.1. Авторизация

| Метод | Эндпоинт | Описание |
|---|---|---|
| POST | `/auth/login` | Вход по логину и паролю, устанавливает сессионную cookie |
| POST | `/auth/logout` | Выход из системы, удаляет сессию |
| GET | `/auth/me` | Получить данные текущего авторизованного пользователя |

#### 3.3.2. Экзамены

| Метод | Эндпоинт | Описание |
|---|---|---|
| GET | `/exams/my` | Список экзаменов текущего пользователя с расчётом статуса |
| POST | `/exams` | Добавить отметку о сдаче экзамена |
| DELETE | `/exams/{id}` | Удалить запись об экзамене |
| GET | `/exams/expiring` | Экзамены всех сотрудников, истекающие в текущем месяце |
| GET | `/exam-types` | Справочник типов экзаменов |

#### 3.3.3. Статусы сотрудников

| Метод | Эндпоинт | Описание |
|---|---|---|
| GET | `/status/my` | История статусов текущего пользователя |
| POST | `/status` | Создать новый статус (больничный / командировка / отпуск) |
| PATCH | `/status/{id}/close` | Закрыть больничный (`end_date` = сегодня) |
| DELETE | `/status/last` | Удалить последний статус текущего пользователя |
| GET | `/status/month` | Статусы всех сотрудников в текущем месяце |
| GET | `/status/current` | Текущая статистика по управлению (процентное распределение) |

### 3.4. Расчётная логика — статусы экзаменов

При получении списка экзаменов backend автоматически рассчитывает:

- **Дата окончания** = `date` + `duration` месяцев.
- **Осталось дней** = `date_end` − сегодня.
- **Статус:**
  - `Действующий` — осталось > 30 дней
  - `Истекает` — осталось от 0 до 30 дней
  - `Просрочен` — осталось < 0

> 💡 Поле `duration` хранится как строка в таблице `exams`. Backend приводит его к `INTEGER` при расчёте.

## 4. Пользовательский интерфейс

### 4.1. Общие принципы UI/UX

- **Мобильная адаптивность** — основной breakpoint 375px (iPhone SE), оптимизация под 390–430px.
- **Тёмная и светлая тема** — переключатель в настройках, предпочтение системы по умолчанию.
- **Bottom Navigation Bar** — основная навигация по нижнему краю экрана.
- **Touch-friendly** — кнопки не менее 44×44 px, отступы между элементами не менее 8px.
- **Загрузочные состояния** — skeleton screens для всех списков, спиннеры для действий.
- **Pull-to-refresh** — обновление списков свайпом вниз.
- **Offline-ready** — отображение последних загруженных данных при отсутствии сети, PWA-кэш.

### 4.2. Экран авторизации

**Отображается, если пользователь не вошёл в систему.**

Элементы:
- Логотип / название системы по центру.
- Поле ввода «Логин» (текстовое поле).
- Поле ввода «Пароль» (скрытый ввод, с кнопкой показать/скрыть).
- Кнопка «Войти» — отправляет `POST /api/v1/auth/login`.
- При ошибке — сообщение под формой: «Неверный логин или пароль».
- После успешного входа — редирект на главный экран.

> 💡 Учётные данные сотрудников (логин и хэш пароля) хранятся в таблице `users`. Требуется добавить поля `login` и `password_hash` при разработке. Сессия хранится на сервере, идентификатор передаётся в cookie.

### 4.3. Главный экран (Home)

**Дашборд с кратким обзором состояния пользователя.**

Элементы:
- Приветствие: «Добро пожаловать, [Имя]!»
- Карточка **«Мои ближайшие экзамены»** — 3 ближайших истекающих экзамена.
- Карточка **«Мой текущий статус»** — активный статус с датами или «На рабочем месте».
- Карточка **«Экзамены истекают в этом месяце»** — количество по всем сотрудникам.
- Быстрые действия: кнопки «+ Отметить экзамен» и «Изменить статус».

### 4.4. Раздел «Экзамены»

#### 4.4.1. Экран «Мои экзамены»
*Маршрут: `/exams/my`*

Список всех экзаменов пользователя, отсортированных по дате окончания (ближайшие первыми).

Каждая карточка содержит:
- Эмодзи + название типа экзамена.
- Дата сдачи в формате `ДД.ММ.ГГГГ`.
- Срок действия до: `ДД.ММ.ГГГГ`.
- Осталось дней: N дней.
- Цветной бейдж статуса: 🟢 Действующий / 🟡 Истекает / 🔴 Просрочен.

FAB «+ Добавить экзамен» в правом нижнем углу.
Пустое состояние: иллюстрация + «Экзамены не найдены. Добавьте первый экзамен».

#### 4.4.2. Экран «Добавить экзамен»
*Маршрут: `/exams/add` (или bottom sheet на мобильном)*

**Шаг 1 — выбор типа экзамена:**
- Сетка кнопок (2 колонки) с эмодзи и названием каждого типа.
- Выбранный тип подсвечивается.

**Шаг 2 — выбор даты сдачи:**
- Нативный date picker или кастомный календарь.
- Нельзя выбрать дату из будущего.
- Предпросмотр: «Срок действия до: [дата]».

**Шаг 3 — подтверждение:**
- Краткая сводка: тип, дата сдачи, дата окончания.
- Кнопка «Сохранить» → `POST /api/v1/exams`.
- После сохранения — возврат на «Мои экзамены», toast «Экзамен добавлен».

#### 4.4.3. Экран «Экзамены на месяц»
*Маршрут: `/exams/expiring`*

Список экзаменов всех сотрудников, истекающих в течение ближайших 30 дней. Группировка по сотрудникам (ФИО + список его истекающих экзаменов).

Фильтр: `Все` / `Критические (< 7 дней)` / `Скоро (7–30 дней)`.

### 4.5. Раздел «Статус сотрудника»

#### 4.5.1. Экран «Мой статус»
*Маршрут: `/status/my`*

Хронологический список всех статусов пользователя. Каждая карточка: эмодзи + тип, период, продолжительность в днях.
Кнопка «Изменить статус» — переход на экран изменения.

#### 4.5.2. Экран «Изменить статус»
*Маршрут: `/status/change` (или bottom sheet)*

Доступные действия:
- 🤒 Больничный
- ✈️ Командировка
- 🏖️ Отпуск
- ❌ Отменить последний статус

**Сценарий «Больничный»:** открывается немедленно текущей датой (`end_date = NULL`). На экране «Мой статус» появляется кнопка «Закрыть больничный» → `PATCH /api/v1/status/{id}/close`.

**Сценарий «Командировка» / «Отпуск»:** выбор даты начала → выбор даты окончания → подтверждение → `POST /api/v1/status`.

**Сценарий «Отменить последний статус»:** диалог подтверждения → `DELETE /api/v1/status/last`.

#### 4.5.3. Экран «Статус на месяц»
*Маршрут: `/status/month`*

Все статусы сотрудников в текущем месяце, сгруппированные по сотрудникам. Сотрудники без статуса отображаются как «На рабочем месте 🟢».

#### 4.5.4. Экран «Текущий статус по управлению»
*Маршрут: `/status/overview`*

- Круговая диаграмма: Работают / Больничный / Командировка / Отпуск.
- Цифры и проценты по каждой категории.
- Раскрывающийся список с ФИО по каждой категории.

## 5. Требования к уведомлениям

### 5.1. Уведомления об истечении экзаменов

Фоновые задачи запускаются ежедневно в **09:00 по московскому времени**.

| Тип уведомления | Поле в БД | Условие |
|---|---|---|
| За месяц до окончания | `month_notification_sent` | `date_end − сегодня` ≤ 30 дней И флаг = false |
| За неделю до окончания | `week_notification_sent` | `date_end − сегодня` ≤ 7 дней И флаг = false |
| В день окончания | `exam_day_notification_sent` | `date_end − сегодня` = 0 И флаг = false |
| Экзамен просрочен | `end_day_notification_sent` | `date_end − сегодня` < 0 И флаг = false |

После отправки уведомления соответствующий флаг устанавливается в `true`.

### 5.2. Push-уведомления в веб-приложении

- Реализуются через **Web Push API** (service worker).
- При первом входе — запрос разрешения на уведомления.
- Текст: «[Тип экзамена] истекает через [N] дней. Сдайте до [дата].»

### 5.3. Еженедельная рассылка статусов

Еженедельно (понедельник, 09:00) формируется сводка по всем сотрудникам с активными статусами. Отображается в разделе «Текущий статус по управлению» как карточка «Последняя рассылка».

## 6. Авторизация

### 6.1. Механизм входа

- Пользователь вводит логин и пароль в веб-форме.
- Backend проверяет соответствие логина и хэша пароля (bcrypt) в таблице `users`.
- При успешной проверке — сервер создаёт сессию и устанавливает cookie в браузере.
- Все защищённые страницы проверяют наличие активной сессии. При отсутствии — редирект на `/login`.
- Кнопка «Выйти» завершает сессию (`POST /api/v1/auth/logout`) и перенаправляет на экран авторизации.

### 6.2. Разграничение прав

| Действие | Сотрудник | Администратор |
|---|---|---|
| Видеть свои экзамены | ✅ | ✅ |
| Видеть чужие экзамены | ❌ | ✅ |
| Добавить свой экзамен | ✅ | ✅ |
| Изменить свой статус | ✅ | ✅ |
| Видеть статусы всех | ✅ (чтение) | ✅ |
| Удалить запись другого | ❌ | ✅ |
| Просматривать статистику | ✅ | ✅ |

> 💡 Роль пользователя хранится в поле `role` таблицы `users`. Требуется добавить при разработке.

## 8. Требования к интерфейсу — компоненты

## 9. Сценарии использования (Use Cases)

### UC-01: Добавление отметки о сдаче экзамена

| Атрибут | Значение |
|---|---|
| Актор | Вошедший в систему сотрудник |
| Предусловие | Пользователь вошёл в систему и находится в разделе «Экзамены» |
| Основной сценарий | 1. Нажать FAB «+ Добавить экзамен». 2. Выбрать тип экзамена. 3. Выбрать дату сдачи. 4. Просмотреть срок действия. 5. Нажать «Сохранить» → `POST /api/v1/exams`. 6. Список обновляется. |
| Постусловие | Запись добавлена в таблицу `exams`, флаги уведомлений = `false` |
| Исключения | Дата в будущем → ошибка «Нельзя выбрать дату из будущего» |

### UC-02: Открытие больничного

| Атрибут | Значение |
|---|---|
| Актор | Вошедший в систему сотрудник |
| Предусловие | Нет активного больничного |
| Основной сценарий | 1. Перейти «Статус» → «Изменить статус». 2. Нажать «Больничный». 3. Система автоматически использует текущую дату. 4. `POST /api/v1/status`. 5. На экране появляется кнопка «Закрыть больничный». |
| Постусловие | Запись в `user_status` с `end_date = NULL` |
| Исключения | Активный больничный уже есть → сообщение об ошибке |

### UC-03: Закрытие больничного

| Атрибут | Значение |
|---|---|
| Актор | Вошедший в систему сотрудник |
| Предусловие | Есть открытый больничный (`end_date = NULL`) |
| Основной сценарий | 1. На экране «Мой статус» нажать «Закрыть больничный». 2. `PATCH /api/v1/status/{id}/close`. 3. `end_date` = сегодня. 4. Карточка обновляется. |
| Постусловие | `end_date` в `user_status` = текущая дата |
| Исключения | — |
